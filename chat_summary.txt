1. Overall backend idea

We converged on this architecture:

Frontend
→ Backend (FastAPI)
→ LLM via OpenRouter for intent parsing
→ MCP server
→ Mappls APIs
→ Backend normalization / filtering / ranking
→ Frontend-ready JSON

Core design decisions:

Use LLM for understanding, not for final ranking.
Keep ranking deterministic in backend.
Maintain session memory for follow-up queries.
Return a frontend-ready response, not raw Mappls output.

6. Radius strategy for Nearby

We decided Nearby should not use a fixed notion of “close.”

Planned strategy:

walkable / near me / nearby → 1000
nearest / closest → 2000
station / airport / railway station → 3000
best / top / popular / broad discovery → 5000
default → 3000

If sparse category:

retry with larger radius later

We added a helper concept in backend:

determine_search_radius(query, intent)

This is to be used in backend/main.py.

7. Distance Matrix inclusion in backend logic

This was the biggest conceptual refinement.

Old design
Nearby → maybe route each place → rank
Improved design
Nearby → shortlist candidates → Distance Matrix → rank
Routing reserved for selected place

Clean role split:

Nearby → candidate retrieval
Distance Matrix → compare many candidates by road distance / ETA
Routing → actual route for chosen result

Planned flow:

query
→ parse intent
→ geocode center
→ determine radius
→ nearby(radius)
→ normalize
→ filter
→ distance matrix enrichment
→ deterministic ranking
→ optional place details enrichment
→ return

We drafted a new backend/main.py concept with:

determine_search_radius(...)
should_use_traffic_matrix(...)
enrich_places_with_distance_matrix(...)

This assumes:

nearby(..., radius=...)
get_distance_matrix(...)

3. Cleaner frontend integration

We improved backend API design by adding a proper POST endpoint.

Before

Backend only had:

GET /query
After

Backend now has:

GET /health
GET /query
POST /query
Why this mattered

POST /query is cleaner for frontend because it supports structured JSON like:

{
  "query": "top cafes near BITS Pilani",
  "session_id": "session-123"
}

instead of stuffing everything into URL query params.

What we changed

In src/backend/main.py:

added QueryRequest
moved all main query logic into process_query(...)
made both GET and POST wrappers call the same function

So:

GET /query stays useful for quick testing
POST /query becomes the real frontend integration endpoint
